# 5.0

* **Release Date:** 2026-07-09

## Overview

This release adds OAuth 2.0 authentication for the Okta management API, moves the project to the maintainer's `BoldlyGrow` namespace, and extends framework support to Laravel v13. The legacy SSWS API token remains fully supported, so existing single-token deployments continue to work without changes.

### New Features

- Add Laravel v13.x support
- Add OAuth 2.0 for Okta authentication using the Client Credentials grant with a `private_key_jwt` client assertion
- Add `ApiToken` class that mints, caches, and returns a short-lived scoped Bearer token
- Add private key resolution from an inline PEM string (`private_key`) or a PEM file path (`private_key_path`)
- Add a per-call `scope` argument to the request methods, required when using OAuth, for least-privilege scoping
- Add an `okta:jwk` Artisan command and a `PublicKeyJwk` action to generate an RSA key pair and convert a public key to a JWK for the Okta service app
- Add configurable `testConnection()` probe endpoint for service apps that are not granted the `okta.orgs.read` scope
- Add authentication-mode aware `401` error messaging

### What's Changed

- **Package Architecture**
    - The Composer package and namespace has been renamed from `provisionesta/okta-api-client` to `boldlygrow/okta-api-client`.
        - **Why:** The project is consolidating under the maintainer's `BoldlyGrow` organization alongside the related audit and actions packages.
    - The PHP namespace changed from `Provisionesta\Okta` to `BoldlyGrow\Okta`.
    - The audit logging dependency changed from `provisionesta/audit` to `boldlygrow/laravel-audit-log`. The imported class changed from `Provisionesta\Audit\Log` to `BoldlyGrow\Audit\Log`.
    - The `provisionesta/okta-api-client`, `gitlab-it/okta-sdk`, and `glamstack/okta-sdk` package names are retained as `replace` aliases for a smoother upgrade.
    - Added `illuminate/console` and an explicit `ext-openssl` requirement to `composer.json` for the new Artisan command and JWT signing.
- **Authentication**
    - Added OAuth 2.0 for Okta as an alternative to the SSWS API token.
        - **Why:** Static SSWS tokens are long-lived org-wide credentials tied to an individual and expire after inactivity. An OAuth service app scopes access to exactly what is needed (for example read-only users, groups, and apps), rotates without human involvement, and pairs with a private key rather than a shared secret.
        - **How:** Configure an Okta API Services app with `Public key/Private key` client authentication, register the public JWK, and grant the required Okta API scopes. Set `OKTA_API_CLIENT_ID` and the key location variables. When `client_id` is present, the client sends a Bearer token; otherwise it falls back to `SSWS`.
        - **Note:** Client ID and client secret is intentionally not supported. Okta only accepts `private_key_jwt` for tokens carrying management API scopes. A client secret only works against a custom authorization server for custom scopes, which cannot read Okta resources.
    - The Bearer token is requested once per unique connection and scope set, encrypted, and cached for 59 minutes. Okta fixes the access token lifetime at one hour.
    - All granted scopes can be requested in a single token, so one token is reused across users, groups, and apps requests. Scopes may also be requested per call.
- **Scopes**
    - Added a `scope` argument to `get`, `post`, `patch`, `put`, and `delete` for per-call scoping, mirroring the Google API client. It is appended as an optional trailing parameter, so existing SSWS calls are unaffected.
    - The scope is required when authenticating with OAuth. There is no config or environment default; omitting it throws a `ScopeException`. The `OKTA_API_SCOPES` variable and the `scopes` config key have been removed.
        - **Why:** The scopes granted to the Okta service app are the hard least-privilege ceiling. Requiring the scope at the call site keeps the requested authority next to the operation that needs it and avoids a broad default that silently widens every token.
    - The SSWS token path ignores the `scope` argument.
    - Each unique scope set is cached separately for the full token lifetime, so per-call scoping mints a few more tokens per hour without shortening caching.
- **Private Key Resolution**
    - The signing key is provided as an inline PEM string (`private_key`) or a PEM file path (`private_key_path`, with a leading `~` expanded). `private_key` takes precedence.
        - **Why:** Keeping the package to two simple key sources avoids conflating a logical secret name with a file path, and keeps the package free of any secrets-manager or cloud SDK coupling.
    - This package intentionally does not integrate with any secrets manager. Fetching the key from a vault or secrets manager is the consuming application's responsibility; the resolved PEM is passed in via the connection array or the published config. See the README for both patterns.
    - The key is resolved only on a token cache miss. The token cache key includes the `kid` and either the file path or a fingerprint of the inline key, so rotating the key takes effect without stale caching.
- **Connections and Configuration**
    - `validateConnection()` now accepts either a `token` (SSWS) or a `client_id` (OAuth). Validated keys are merged over the source connection array so OAuth keys provided at runtime are preserved through to `ApiToken`.
    - Added configuration keys: `client_id`, `key_id`, `private_key`, `private_key_path`, and `test_uri`. Scope is passed per request, not configured.
- **API Requests**
    - `testConnection()` still probes `org` by default but now honors `OKTA_API_TEST_URI`.
        - **Why:** The `org` endpoint requires the `okta.orgs.read` scope. A service app granted only users, groups, and apps read would otherwise fail the connection test despite having valid credentials.
- **Error Handling**
    - The `401` exception message is now tailored to the active authentication mode. Under OAuth it points to token expiry, scope grants, and JWK registration rather than SSWS token rotation.

### Upgrade Steps

- Update `composer.json` to require `boldlygrow/okta-api-client:^5.0` and run `composer update`.
- Perform a find and replace across your code base from `Provisionesta\Okta` to `BoldlyGrow\Okta`.
- If you reference the audit logger directly, update `Provisionesta\Audit` to `BoldlyGrow\Audit`.
- Re-publish or update the config file so the new OAuth keys are present: `php artisan vendor:publish --tag=okta-api-client`.
- Existing SSWS deployments require no functional change. Keep `OKTA_API_URL` and `OKTA_API_TOKEN` as-is.
- To move a connection to OAuth:
    - Create an Okta API Services app with `Public key/Private key` client authentication and register your public JWK.
    - Grant the app the Okta API scopes it needs (for example `okta.users.read`, `okta.groups.read`, `okta.apps.read`) and assign a least-privilege admin role.
    - Set `OKTA_API_CLIENT_ID` and `OKTA_API_KEY_ID`. Scopes are passed per call via the `scope` argument, not an environment variable.
    - Provide the private key with `OKTA_API_PRIVATE_KEY_PATH` (a PEM file) or `OKTA_API_PRIVATE_KEY` (an inline PEM), or pass `private_key` in the connection array. For production, resolve the key from your secrets manager in your own code and pass it in the connection array.
    - If the service app is not granted `okta.orgs.read`, set `OKTA_API_TEST_URI` to an endpoint it can reach.
- This is not an exhaustive list. Review the `README.md` for the latest syntax and usage guidelines.
